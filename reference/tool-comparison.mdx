## Recommended Stacks

Choose the stack that matches your team's infrastructure and workflow.

<AccordionGroup>
  <Accordion title="Enterprise Atlassian Cloud">
    Best for large teams on Atlassian Cloud with managed infrastructure:

    - **Rovo MCP** — native JIRA + Confluence integration, zero config for Cloud tenants
    - **Tabnine Enterprise** — zero data retention, enterprise security, JetBrains native
    - **Acendas Suite** — 179 Atlassian-specific tools for deep ticket and doc automation
  </Accordion>

  <Accordion title="Self-hosted JIRA + GitLab CE">
    Best for teams running JIRA Data Center and GitLab Community Edition on-premises:

    - **Tabnine Enterprise** — JIRA Data Center support, keeps all data on-prem
    - **JIRA GitLab Helper** — purpose-built for the JIRA + GitLab CE combination; handles MR linking, branch detection, and status transitions out of the box
    - **Forge Orchestrator** — works with any self-hosted Git provider, including GitLab CE and Gitea
  </Accordion>

  <Accordion title="Open-source / Indie Dev (Cursor / Claude)">
    Best for solo developers or small teams using Cursor or Claude Code with open tooling:

    - **Task Master AI** — file-based task trees, fully open source, MCP-native
    - **Rovo MCP** — connects to Atlassian Cloud if you use it; otherwise use the GitLab MCP server for self-hosted Git context
    - **Markplane** — keeps tasks in Markdown inside the repo, no external services needed
  </Accordion>

  <Accordion title="Multi-agent Power Users">
    Best for teams running parallel AI agents on large epics:

    - **Forge Orchestrator** — file locking, knowledge flywheel, multi-agent TUI
    - **CLEO** — persistent agent memory across sessions
    - **Atlassian Rovo MCP** — provides both agents shared access to the same JIRA + Confluence data
  </Accordion>

  <Accordion title="This Team: JIRA + Confluence + IDEA + Self-hosted Git">
    The recommended stack for teams using IntelliJ IDEA, JIRA (Data Center or Cloud), Confluence, and a self-hosted Git platform (GitLab CE or Gitea):

    - **Tabnine Enterprise** — JIRA Data Center support, IntelliJ IDEA native plugin, zero data retention; or **Rovo MCP** if your JIRA tenant is on Atlassian Cloud
    - **JIRA GitLab Helper** — MR-to-ticket linking, branch-based JIRA key detection, and auto-close for GitLab CE
    - **Forge Orchestrator** — multi-agent parallelism when working on large epics (`forge run` / `forge ship`)
    - **CLEO** — persistent memory so agents retain design decisions across sessions and IDE restarts
    - **Acendas Suite** — 179-tool MCP server for deep Atlassian automation when you need bulk operations across JIRA and Confluence
  </Accordion>
</AccordionGroup>

## Decision Flowchart

Work through these questions in order to narrow your tool selection:

1. **Are you on Atlassian Cloud?**
   → Yes: Start with **Rovo MCP** — it requires no additional setup for Cloud tenants.

2. **Are you on JIRA Data Center (self-hosted)?**
   → Yes: Use **Tabnine Enterprise** — it natively supports DC and keeps data on-prem.

3. **Using GitLab CE for Git?**
   → Yes: Add **JIRA GitLab Helper** — it handles MR linking, branch detection, and JIRA transitions for GitLab CE out of the box.

4. **Need persistent agent memory across sessions?**
   → Yes: Add **CLEO** or **Fenix** — both maintain a knowledge store that survives IDE restarts and project switches.

5. **Need multi-agent parallelism on large epics?**
   → Yes: Add **Forge Orchestrator** — it manages file locking, agent scopes, and produces a single reviewed MR from multiple agents' work.

## JIRA Slash Command Integrations

Some tools extend JIRA's own interface with slash commands rather than operating as IDE plugins or MCP servers.

**jira-claude-code-integration** adds `/jira-task` commands directly inside JIRA tickets. Once installed, you can open any ticket and run `/jira-task implement` or `/jira-task write-tests` to send the ticket context to Claude Code and receive a branch, commits, and an MR draft without leaving the JIRA interface. This is useful for PMs or tech leads who want to trigger implementation from JIRA without switching to the IDE. See [JIRA GitLab Helper](/tools/jira-gitlab-helper) for the complementary MR-linking side of this workflow.
