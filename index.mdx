## The AI Development Lifecycle

Every task in the KN Group workflow moves through a predictable, automated sequence. The steps below show the canonical path from a new JIRA issue all the way to a closed ticket in production.

<Steps>
  <Step title="JIRA Issue">
    A ticket is created with acceptance criteria and story points. This is the single source of truth that drives everything downstream.
  </Step>
  <Step title="Confluence Context">
    Architecture Decision Records, design docs, and runbooks are linked to the ticket so the AI agent receives full project context before touching any code.
  </Step>
  <Step title="IDE + LLM">
    An AI coding agent (Claude, Codex, or equivalent) reads the JIRA ticket and Confluence docs via MCP, then generates or modifies code directly inside IntelliJ IDEA or VS Code.
  </Step>
  <Step title="Git MR">
    The agent opens a Merge Request on your self-hosted Git server with the JIRA issue key in the branch name, commit message, and MR description.
  </Step>
  <Step title="CI/CD Pipeline">
    Automated tests, linters, and security scans run on every push. On merge to the main branch, the pipeline triggers a deployment.
  </Step>
  <Step title="JIRA Closed">
    The CI/CD pipeline calls the JIRA API to transition the originating ticket to **Done** and notifies stakeholders — no manual status updates required.
  </Step>
</Steps>

<Note>
  **MCP is the universal integration layer.** Model Context Protocol (MCP) servers expose JIRA, Confluence, and Git to any compatible IDE or LLM agent over a single, standardised interface. You configure it once and every tool in your stack speaks the same language. See the [MCP Protocol](/mcp-protocol) page for configuration details.
</Note>
